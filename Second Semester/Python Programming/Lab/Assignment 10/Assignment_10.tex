\section{Theory}

\subsection{Threads}
\begin{quotation}
    A Thread is a single sequential flow of control within a program. The concept of threads is similar to that of processes. 
\end{quotation}

\begin{itemize}
    \item Threads are sometimes called lightweight processes because they have their own stack but can access shared data. Because threads share the same address space as the process and other threads within the process, the operational cost of communication between the threads is low, which is an advantage. The disadvantage is that a problem with one thread in a process will certainly affect other threads and the viability of the process itself.
    
    \item Threads allow the application to be more responsive to the user. Threads can also take advantage of multiprocessor architectures as processes can be assigned to different processors. The operating system manages the threads within a process. Also see multitasking, multiprocessing, multithreading.
    
    \item A thread is a dispatchable unit of work. Threads are sometimes called lightweight processes. Both processes and threads provide an execution environment, but creating a new thread requires fewer resources than creating a new process. Threads exist within a process — every process has at least one. Threads share the process's resources, including memory and open files. This makes for efficient, but potentially problematic, communication.
\end{itemize}

\textbf{Example: }

\begin{lstlisting}[language=python]
t1 = threading.Thread(target=print_cube, args=(10,))
t2 = threading.Thread(target=print_square, args=(10,))
t1.start()
t2.start()
\end{lstlisting}

\subsection{Process}

\begin{quotation}
    A Process is a collection of one or more threads that share the same virtual memory, code segment, data segment, and operating-system resources, such as open files and signals.
\end{quotation}
\begin{itemize}
    \item 
    \item A process is controlled by the operating system, which allocates resources and regulates execution. Threads are not controlled by the operating system, but rather are driven by plain Java code, with some help from the JVM.
    
    \item A process is an execution of a program, but a thread is a single execution sequence within a process. A process can contain multiple threads. A thread is sometimes called a lightweight process.
    
    \item A process is an instance of a program running in a computer. It is close in meaning to task , a term used in some operating systems. In UNIX and some other operating systems, a process is started when a program is initiated (either by a user entering a shell command or by another program). Like a task, a process is a running program with which a particular set of data is associated so that the process can be kept track of. Each process can be assigned a certain priority. Because the operating system must manage the allocation of resources to processes, it must also keep track of where each process resides in memory and where and how much memory is available for each process. This activity is called process management.
\end{itemize}

\subsection{Multithreading}

\begin{quotation}
    Multithreading is the ability of a program or an operating system process to manage its use by more than one user at a time and to even manage multiple requests by the same user without having to have multiple copies of the programming running in the computer.

\end{quotation}

\begin{itemize}
    \item It is a process of executing multiple threads simultaneously.
    \item It enables a program to perform more than one task at a time, by using the maximum CPU time.
    \item The main benefit of multithreading is that it enables a program to operate more efficiently by doing multiple things at the same time, rather than one at a time.
    \item Under the hood, most operating systems use preemptive multitasking, which means each running program is allowed to run for a short time before another program gets a turn. This creates the illusion that multiple programs are running simultaneously, even though only one program is running at a time.
\end{itemize}

\subsection{Threading Module}

\begin{quotation}
    The threading module builds on the low-level features of thread to make working with threads even easier and more pythonic. It offers a higher-level interface than the thread module: threading.Thread is a class that represents a thread of control.
\end{quotation}

\begin{itemize}
    \item The threading module provides a number of functions and classes to manage threads.
    \item It allows to create simple threaded programs.
    \item It provides a way to run multiple functions at the same time and also provides a mechanism to synchronize the threads.    
\end{itemize}
\subsection{Functions of the Threading Module}

There are many functions in the threading module. Some of them are listed below:

\begin{enumerate}
    \item \textbf{threading.activeCount()}: Returns the number of thread objects that are active.
    \item \textbf{threading.currentThread()}: Returns the number of thread objects in the caller's thread control.
    \item  \textbf{threading.enumerate()}: Returns a list of all thread objects that are currently active.
    \item  \textbf{threading.LineLock()}: This function is used to block the current thread until the lock is released.
    \item  \textbf{threading.Lock()}: This function is used to acquire a lock, and it is used to release the lock.
    \item \textbf{threading.Condition()}: This function is used to wait until notified or until a timeout occurs.
    \item  \textbf{threading.Semaphore(value=1)}: This function is used to release the semaphore counter.
    \item \textbf{threading.Thread(group=None, target=None, name=None, args=(), kwargs={}, *, daemon=None)}: This function is used to represent an activity that is run in a separate thread of control.
    \item  \textbf{threading.Timer}: This function is used to run a function after a specific interval of time.
\end{enumerate}
Here are some functions used to manage threads:
\textbf{Here are some functions used to manage threads:}

\begin{enumerate}
    \item \textbf{threading.start()}: This function is used to start a thread.
    \item \textbf{threading.run()}: This function is used to run a thread.
    \item \textbf{threading.join([timeout])}: This function is used to wait until the thread terminates.
    \item \textbf{threading.isAlive()}: This function is used to check whether a thread is still executing.
    \item \textbf{threading.getName()}: This function is used to return the name of the thread.
    \item \textbf{threading.setName()}: This function is used to set the name of the thread.
\end{enumerate}

\section{FAQ}
\begin{enumerate}
    \item \textbf{What's the difference between a process and a thread ?}\\
    The difference between a process and a thread is that a process is an executing program with its own address space, whereas a thread is a single execution sequence within the process. A process can contain multiple threads. A thread is sometimes referred to as a lightweight process.
    Some other differences between a process and a thread include:

    \begin{itemize}
        \item Processes are isolated by default whereas threads share memory.
        \item  Processes have their own address space, whereas threads share the address space.
        \item  Process creation is slow, whereas thread creation is fast.
        \item  Processes are independent, whereas threads are not independent.
        \item  Process termination is slow, whereas thread termination is fast.
        \item  Processes don't share memory with other processes, whereas threads share memory with other threads of the same process.
        \item  Processes are heavyweight operations, whereas threads are light weight operations.
        \item  Process switching needs interaction with the kernel, whereas thread switching does not need interaction with the kernel.
        \item  Processes are independent, so if one process is blocked, it doesn't affect the execution of other processes. Threads share the same address space, so if one thread is blocked, it affects the execution of the other threads of the same process.
        \item  Processes don't need to be synchronised, whereas threads need to be synchronised.
    \end{itemize}
    \item \textbf{When should use Multithreading.}\\
    
    We should use Multithreading in the following cases:

    \begin{itemize}
        \item  When we want to perform multiple tasks at a time so that we can reduce the time taken to perform the tasks.
        \item  When we want to perform multiple I/O tasks at a time so that there is no wastage of CPU cycles.
        \item  When we want to perform multiple tasks that are not dependent on each other.
    \end{itemize}

    
\end{enumerate}
